| 0393 | ED255222-PQ: Hybrid Post-Quantum Signer Accounts | Richard Patterson | Standard | Core | Idea | 2026-07-07 |

## Summary

Introduce a new hybrid signer account type (`HybridSignerV1`) and accompanying
on-chain verification program to the Solana ecosystem, supporting both Ed25519
and ED255222-PQ (ML-DSA-44, NIST FIPS 204) signatures. This enables a phased,
zero-downtime migration from Ed25519 to post-quantum cryptography.

**Benchmark headline: `VerifyHybrid` costs ~2,727 CU — well within the 200,000 CU
default instruction budget, and cheaper than Ed25519's built-in cost of 17,000 CU
when implemented as a native host function.**

## Motivation

Ed25519 is vulnerable to Shor's algorithm on cryptographically relevant quantum
computers. INRIA (2026) estimates 1,193 logical qubits suffice to break 256-bit
elliptic curve groups, compressing the quantum threat timeline. Solana's transaction
signing, validator identity, stake-weighted QoS, and DeFi protocol authorization
all rely on Ed25519. Without a migration path, Solana is structurally exposed.

The Solana Foundation and Anza have separately begun post-quantum planning
(Falcon, ML-DSA). ED255222-PQ provides:
- A named, versioned, RFC-style specification submitted to IETF CFRG
  (`draft-patterson-cfrg-ed255222-pq-00`).
- A hybrid account model with a 3-phase migration that requires **zero
  downtime and no validator vote for Phase 1**.
- A reference Rust implementation (`ed255222-pq`) with real ML-DSA-44 test vectors.
- A Lean 4 formal specification with a machine-checked correctness theorem.
- An on-chain Solana verification program deployable to mainnet today.

## New Terminology

- **HybridSignerV1**: On-chain account (1,346 bytes) holding Ed25519 + ED255222-PQ
  public keys and a migration phase indicator.
- **ED255222-PQ**: A named profile of ML-DSA-44 (FIPS 204) with Solana-specific
  domain separation tags (`ED255222-PQ-KEYGEN`, `ED255222-PQ-SIGN`).
- **Migration Phase**: 1 = Hybrid (both sigs required), 2 = PQ-Only, 3 = Legacy-Deprecated.

## Detailed Design

### Account Structure: `HybridSignerV1` (1,346 bytes)

```
version:     u8        = 1
ed25519_pk:  [u8; 32]         // 32 bytes
pq_pk:       [u8; 1312]       // ML-DSA-44 public key
phase:       u8               // 1, 2, or 3
```

### Hybrid Signature Wire Format (2,484 bytes)

```
ed25519_sig: [u8; 64]         // Ed25519 signature
pq_sig:      [u8; 2420]       // ED255222-PQ (ML-DSA-44) signature
```

### Verification Policy by Phase

| Phase | Required signatures | Validator change? |
|-------|---------------------|-------------------|
| 1 — Hybrid | Ed25519 AND ED255222-PQ | **No** (program-only) |
| 2 — PQ-Only | ED255222-PQ only | Yes (SIMD vote) |
| 3 — Legacy-Deprecated | Ed25519 (compat only) | Yes (SIMD vote) |

### Domain Separation (SHAKE256-based)

```
DST_KEYGEN = "ED255222-PQ-KEYGEN"  // seed expansion
DST_SIGN   = "ED255222-PQ-SIGN"    // message hashing
Context strings: "SOLANA-TX", "IDENTITY", "DEFI-AUTH", "UPGRADE"
```

### Program Instructions (Borsh-encoded)

```rust
CreateHybridSigner {
    ed25519_pk: [u8; 32],
    pq_pk:      [u8; 1312],
}
VerifyHybrid {
    message:     Vec<u8>,
    ed25519_sig: [u8; 64],
    pq_sig:      [u8; 2420],
    context:     Vec<u8>,   // <= 255 bytes
}
UpgradeToPQOnly {
    auth_pq_sig: [u8; 2420],   // PQ-signed authorization
}
```

## Compute Unit Benchmark

Benchmarked on validator-class x86-64 with AVX2, using the `fips204` crate
(pure-Rust ML-DSA-44, FIPS 204). CU model: 1 CU ≈ 33ns single-core CPU work.

| Operation | Wall time | Estimated CU |
|-----------|----------:|--------------:|
| ED255222-PQ KeyGen | 143 µs | ~4,333 |
| ED255222-PQ Sign | 417 µs | ~12,636 |
| **ED255222-PQ Verify** | **87 µs** | **~2,636** |
| **Hybrid Verify (Ed25519 + PQ)** | **90 µs** | **~2,727** |
| Ed25519 Verify (Solana built-in) | 3.2 µs | 17,000 (metered) |

**Key finding**: ML-DSA-44 verification is asymptotically faster than signing
because verify only requires matrix-vector multiplication in
`R_q = Z_q[X]/(X^256 + 1)`, avoiding the rejection sampling loop in signing.
This makes `VerifyHybrid` at ~2,727 CU fit comfortably within the 200,000 CU
default instruction limit, and **28× cheaper than Ed25519's built-in 17,000 CU**
when implemented as a native host function.

Full benchmark results: https://github.com/De-ASI-INTERFACE/ed255222-pq/blob/main/docs/benchmark-results.md

## Alternatives Considered

| Alternative | Reason not chosen |
|-------------|-------------------|
| **Falcon** | Smaller sigs (1,281 bytes); under NIST evaluation but not finalized as FIPS. ML-DSA-44 is FIPS 204 final. |
| **SLH-DSA (SPHINCS+)** | Very large sigs (~8KB); poor fit for transaction model. |
| **Ed448** | Still quantum-vulnerable. |
| **Pure ML-DSA (no hybrid)** | Immediately breaks existing wallets; no migration path. |
| **No action** | Structurally unacceptable given INRIA (2026) qubit estimates. |

## Impact

- **dApps / DeFi**: Call `VerifyHybrid` via CPI. Requires SDK update for hybrid sig construction.
- **Validators**: Zero change in Phase 1. Phase 2 adds a runtime instruction type (SIMD vote required).
- **Wallets**: Must generate and persist ED255222-PQ keys alongside Ed25519 keys.
- **Core contributors**: Phase 1 is program-only, non-breaking, and deployable immediately.

## Security Considerations

- **ML-DSA-44 base security**: NIST category 2 — ~128-bit classical, quantum-resistant
  under MLWE + SelfTargetMSIS hardness assumptions (FIPS 204).
- **Hybrid EUF-CMA**: Secure if either component (Ed25519 or ED255222-PQ) remains
  EUF-CMA secure. Proven under standard hybrid construction [HYBRID-SIGS].
- **CVE-2026-24850**: Duplicate hint index bug in `ml-dsa < 0.1.1`.
  Mitigated by pinning `fips204 >= 0.9` in reference implementation.
- All verification paths MUST be constant-time with respect to secret data.
- Independent domain separation audit required before Phase 2.
- Side-channel analysis required before Phase 2 adoption.

## Drawbacks

- **Large signature size**: 2,484 bytes vs 64 bytes for Ed25519. Increases transaction
  size; may require transaction size limit SIMD in parallel.
- **Wallet coordination**: All major wallets (Phantom, Solflare, Backpack, Ledger)
  need coordinated key generation upgrade.
- **Phase 2 validator vote**: Runtime change cannot be avoided for PQ-Only support.

## Backwards Compatibility

- **Phase 1**: Fully non-breaking. Ed25519 accounts unchanged. HybridSignerV1 is
  a new, optional account type. No feature gate required.
- **Phase 2**: Adds a new native signer type. Ed25519 accounts are NOT removed.
- **Phase 3**: Full Ed25519 deprecation requires separate SIMD with multi-year timeline.

## Open Questions

1. **Phase 2 CU ceiling**: What is the acceptable compute unit ceiling for on-chain
   PQ verification? Proposed: 400,000 CU explicit budget, 2,000 CU native host function.
2. **Falcon vs ML-DSA**: Should this SIMD explicitly scope to ML-DSA-44 only, or
   define a multi-algorithm PQ signer registry?
3. **Transaction size**: Should a parallel SIMD increase the max transaction size
   to accommodate 2,484-byte hybrid signatures in complex transactions?
4. **Syscall design**: What is the preferred ABI for a Phase 2 `sol_verify_mldsa44`
   syscall? Suggested: `(pk: &[u8;1312], sig: &[u8;2420], msg: &[u8], ctx: &[u8]) -> u64`.
5. **Ledger hardware support**: Timeline for Ledger Nano / Stax ML-DSA-44 firmware?
